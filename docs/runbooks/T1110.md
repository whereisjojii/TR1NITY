# Brute force (T1110)

An adversary is trying many credentials against an authentication
service. The Wazuh ingestor surfaces failed-login bursts; the firewall
ingestor will often see the same source IP touching `:22`, `:3389`,
`:445`, or HTTP login pages.

## Triage (first 5 min)

- Confirm this is **not** a known scanner from the operator
  whitelist (Layer 1) — the FP score will already be ≥0.95 if so.
- Look at `members[*].source_ip` and `host.name`. A single source IP
  attacking many users → password spraying (see T1110.003). Many
  source IPs attacking one account → credential stuffing.
- Pull the rule that fired (`sigma_matches`). The rule packs ship with
  thresholds, so a hit here means the threshold was breached.

## Investigation

- Did **any** member event flip from `failed` to `accepted`? That's
  the worst-case signal — pivot to T1078 (Valid accounts) and treat
  the affected account as compromised.
- Check `intel_hits` for known abusive IPs.
- Cross-reference the source IP against your VPN concentrator: real
  user from a new geography, or true external?

## Containment

- Add a Layer-3 suppression **only** if you've confirmed the source is
  a legitimate batch job — TTL of ≤7 days while you fix the alerter.
- For external sources: block at the firewall (egress + ingress) and
  add the IP to your blocklist feed.
- For internal sources: isolate the host.

## Eradication & lessons

- Reset credentials for any successful logon during the burst.
- Verify MFA enrolment for the targeted account(s).
- If the rule fired far above its threshold, raise the threshold
  _only_ after writing a postmortem — don't silently desensitise.
