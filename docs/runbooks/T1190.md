# Exploit public-facing application (T1190)

Web exploit attempts (SQLi, RCE, path traversal, deserialisation)
against an internet-facing app. The WAF ingestor surfaces these via
ModSecurity audit logs; SIGMA rules `waf-sqli` and `waf-path-traversal`
ship in the rule pack.

## Triage

- Confirm the request hit a **production** app, not a honey-pot or
  staging surface.
- Pull the exact request body / URL from `members[*].message`. A
  partial match against a known WAF rule (e.g. `949110`) is _not_ a
  successful exploit by itself.

## Investigation

- Did the WAF block the request (`5xx`, `403`)? If yes and there's no
  follow-up activity, this is mostly noise — but still worth keeping
  the alert in the queue for trend-spotting.
- If not blocked: pull the next 30 minutes of requests from the same
  source IP. Look for tool fingerprints (User-Agent: sqlmap, nuclei,
  nikto).
- Search the application logs for evidence of code execution: spawned
  shells, child processes of the web worker, outbound connections.

## Containment

- Add a temporary WAF rule blocking the offending source / pattern.
- If exploitation is confirmed: snapshot the host and isolate it from
  the network before doing anything else.

## Eradication & lessons

- Patch the exploited CVE; capture the version drift in your asset
  inventory.
- Rotate any secrets the application had access to (DB creds, service
  account tokens).
- File a postmortem: time-to-detection vs. time-to-block matters here.
