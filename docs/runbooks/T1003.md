# OS credential dumping (T1003)

Adversary reads `LSASS`, `SAM`, `/etc/shadow`, or memory secrets. This
is the precursor to lateral movement (T1078, T1021) and is one of the
highest-priority alerts in the queue.

## Triage

- Confirm the alert isn't from a legitimate scanner (BloodHound,
  CrackMapExec, ADExplorer) authorized for an active engagement.
- Process trees pointing at `lsass.exe`, `procdump.exe -ma lsass.exe`,
  `mimikatz`, or remote SAM hive copy → almost always real.

## Investigation

- Walk the source host **immediately**. Memory snapshot for forensics.
- Look at the source account's recent privileged activities. Any
  `Domain Admins` or service-account access?
- Check whether any cached credentials may now be in adversary
  hands — every cached credential on the box should be considered
  compromised.

## Containment

- Isolate the host from the network. Do not power it off — keep
  forensics alive.
- Force-rotate credentials for every account that has logged onto the
  box in the past 30 days.

## Eradication & lessons

- Reimage. Don't trust the OS once LSASS has been touched.
- Roll Kerberos krbtgt twice.
- Walk the kill chain: how did the adversary land here?
